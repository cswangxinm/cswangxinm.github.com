#####1. Import, from
当import时，python会自动执行import的程序。模块导入的过程：

  1. Find the module’s file.
  2. Compile it to byte code.
  3. Run the module’s code to build the objects it defines.

注意上面的过程只在第一次导入时发生。以后导入后，python仅仅在内存中寻找对应的模块。python将所有模块保存在一个表中：sys.modules。导入模块时，python会检查该模块是不是在其中。如果不在其中，会发生上面的3个步骤。在编译模块之前，python检查文件的时间戳。如果.pyc文件不比源文件的.py文件老，则会忽略第二步。还有一点要注意的是，import会忽略导入模块的路径名。为了区别导入时的运行行为和作为主文件运行的运行行为。可以使用__name__ 和 __main__ 来限定。

#####2. 寻找路径
python寻找路径的过程是：

  1. The home directory of the program(当前目录)
  2. PYTHONPATH directories(if set)(在windows系统下可以增加一个PYTHONPATH环境变量)
  3. Standard library directories
  4. The contents of any .pth files(if present)(可以在cite-packages下建立.pth文件。添加目录)

第4点添加文件会比较方便。但是.pth文件要放在site-packages.这个目录相对于python的安装路径。而不是当前目录。

最后，可以使用sys.path来查看ptyon的search path。

#####3. from…*
这种方式在Python 3.0中只能在top level中使用。在函数里面是不能使用的。
from方法可以单独导出某些函数或变量。下面的两种方式是等价的。

from module import name1, name2
the same as:

#import module
name1 = module.name1
name2 = module.name2
del module

尽管通过from可以导入特定的函数或变量。但是用import导入模块会使得程序更加规范和易读。有时，使用import是必须的。比如下面的代码：

from M import func
from N import func
func() #Calls N.func

如果嫌模块的变量太复杂。可以使用as关键字引入别名。
import Module as M
事实上，模块被导入后，namespaces是存放在dictionary对象中的。我们可以通过__dict__属性访问。

import module1 as mo
print mo.__dict__.keys()
print mo.__file__ #file infomation
print mo.__name__ #file name

#result:
['__builtins__', '__file__', '__package__', 'x', '__name__', '__doc__']
D:\project\python\learn\module1.pyc
module1

#####4. reload
reload 现在紧紧对Python写的module有效。对于比如C语言写的module,不能reload。用from展开的变量或者函数不能使用reload恢复最新的值。

#####5. module packages
假设我们在dir0目录。我们要访问dir0/dir1/dir2 目录下的mod.py文件。import时，需要用.区分目录。每个目录下必须要有__init__.py文件。但是目录dir0里面不需要使用__init__。下面是一个例子。

import dir1.mod

当调用目录里的模块时，python会首先执行__init__进行一些初始化工作。所以在init文件里面可以进行一些初始化工作。对于from *的导入语句，可以在init文件中通过使用__all__来指定哪些子模块是需要导入的。通常使用from语句来导入package里的模块。这样访问起来就会方便很多。

在python2.x中，会首先搜索当前目录。也就是相对的path。而在python3.x中，要使用from . dir1.dir2 import module来导入模块。不然python总是会忽略相对路径。这里还要注意。上面所说的特性在3.0中仅仅适合from插入。如果是import modname. 3.0也会先搜寻相对路径。

还要注意的一点是：包含相对路径的import脚本不能直接运行，只能作为module被引用。原因正如手册中描述的，相对路径是按照module的路径，但如果直接运行脚本，该脚本就变成’main‘了。也就不是原来的name了。此时，相对路径不再是原来的路径了

ValueError: Attempted relative import in non-package

这里还要注意一个问题：顶层目录(main文件所在目录)并不是一个package.也就意味着下面的调用方式是错误的。

  • start.py
  • sub/module1.py
  • module2.py
    如果在start中import module1.py。而module1.py调用module2.py。module2.py可能这么些：from .. import module2。这样写会得到下面的错误

ValueError: Attempted relative import beyond toplevel package

  • start.py
  • sub/sub/module1.py
  • sub/module2.py
    此时运行start.py函数可以正确执行。

模块导入过程总结

  • 简单导入。从sys.path从左到右匹配。
  • 使用from . 导入只在相对目录中找。不在sys.path中找。

#####6. Data hiding
有时，需要对import的模块的某些函数和变量进行隐藏。我们可以通过：

  1. 在变量前加下滑线来避免from方式的导入。但是这对import无效
  2. 在模块的顶层使用__all__关键字。此后，from * 只拷贝在all关键词中出现的对象。(同样对import是无效的)

#####7. Enabling Future Language Features

from __future__ import featurename

#####8. 自定义search path
程序运行后，可以修改sys.path来自定义寻址路径。这样你就可以动态地修改search地址了。比如：

>>> import sys
>>> sys.path
['', 'C:\\users', 'C:\\Windows\\system32\\python30.zip', ...more deleted...]
>>> sys.path.append('C:\\sourcedir')         # Extend module search path
>>> import string                            # All imports search the new dir last

要注意的是，这种修改只对运行中的程序有效。当程序推出后。不会被保存到PYTHONPATH中。
此外，动态加载可以通过exec来操作。使用exec的局限性是每次运行exec，都要重新编译模块。如果使用__import__,会使得程序更快一点。reload并不嵌套reload。

trick: 在可以在字符串前用r指定为路径。

#####9. code执行顺序
top-level的代码会从上到下执行。而函数里的代码在被调用时才被执行。比如：

func1()                           # Error: "func1" not yet assigned
def func1():
    print(func2())                # Okay: "func2" looked up later
func1()                           # Error: "func2" not yet assigned
def func2():
    return "Hello"
func1()                           # Okay: "func1" and "func2" assigned

#####10. 一些注意点

  • 下面的代码：
    ```python
    #nested1.py
    x = 99
    def printer(): print(x)

#nested2.py
from nested1 import x, printer
x = 88
printer()

#result:
99
```
改变x，改的是local的变量。而不是导入模块的变量。所以拷贝的是名字而不是链接。
